### Finding 1 — `DispatchDefaults` is exported but unused after the dispatch-time revalidation fix

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/repo.rs:18-43` (definition + `Default` impl), `crates/core/src/lib.rs:22` (re-export) |
| **Severity** | Low |
| **Category** | Idiom |

**Problem**

The previous review's Finding 1 fix replaced `DispatchDefaults` with `&DispatchConfig` as the validation context for `extract_callback_target`. After that change, `DispatchDefaults` has no remaining call sites in the workspace (`grep -rn DispatchDefaults` shows three hits — definition, `Default` impl, and the public re-export — and zero callers). It's still pulled in via `pub use repo::{DispatchDefaults, …}`, which surfaces dead public API and means anyone reading `lib.rs` will assume it's load-bearing. CLAUDE.md is explicit that unused leftovers should be deleted rather than kept around for "future use".

**Context** (surrounding code as it exists today)


```rust

```

**Recommended fix**

Delete the struct, its `Default` impl, the doc comment, and the re-export.

```rust

```

(and remove the `DispatchDefaults` struct + `Default` impl block from `crates/core/src/repo.rs` entirely.)

**Why this fix**

The struct duplicates fields that already live on `DispatchConfig` and is no longer referenced by production code or tests. CLAUDE.md: "If you are certain that something is unused, you can delete it completely." Removing it shrinks the public surface and prevents future contributors from wiring the wrong defaults source.

### Finding 2 — `extract_callback_target` parses every sibling callback to look up one

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/repo.rs:728-758` |
| **Severity** | Low |
| **Category** | Performance |

**Problem**

`extract_callback_target` calls `parse_callbacks(callbacks, config)`, which fully parses *every* callback in the array into a `CallbackTarget` (cloning the `headers: HashMap<String, String>`, building `backoff: Vec<Duration>`, etc.) before the function discards all but the matching entry. With `batch_size = 50` and `max_callbacks_per_event = 32` (defaults), the dispatch hot path can build up to 50 × 32 = 1,600 fully-materialised `CallbackTarget`s per polling cycle to keep 50 of them. This is an O(N²) scan over total callbacks per event, where O(N) suffices — and the wasted allocations are exactly the heaviest fields (HashMap rehashes, Vec growth) of the type.

The defense-in-depth motivation is correct (re-validate URL scheme, headers, host, signing requirement at dispatch time), but the validation only needs to run on the matching entry plus any cross-callback checks (`max_callbacks_per_event`, duplicate names).

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/repo.rs lines 722-758
fn extract_callback_target(
    callbacks: &serde_json::Value,
    callback_name: &str,
    config: &DispatchConfig,
) -> Result<CallbackTarget> {
    // Must be an array — return a dedicated error so the poison-pill guard logs it clearly.
    if !callbacks.is_array() {
        return Err(crate::error::Error::InvalidData(
            "callbacks is not an array".to_string(),
        ));
    }

    let parsed = crate::callbacks::parse_callbacks(callbacks, config);

    // If the named callback appears in `invalid`, it failed structural revalidation.
    if let Some((_, reason)) = parsed.invalid.iter().find(|(n, _)| n == callback_name) {
        return Err(crate::error::Error::InvalidData(format!(
            "callback '{callback_name}' failed structural revalidation at dispatch time: {reason}"
        )));
    }

    parsed
        .valid
        .into_iter()
        .find(|t| t.name == callback_name)
        .ok_or_else(|| {
            crate::error::Error::CallbackTargetMissing(format!(
                "callback '{callback_name}' not found in event callbacks array"
            ))
        })
}
```

**Recommended fix**

Expose a narrower helper from `callbacks` that runs the same per-callback validation (URL scheme, headers, signing requirement, mode/timeout pairing, …) on a single entry, plus the cheap `max_callbacks_per_event` array-level check, and return either the parsed target or the rejection reason.

```rust
// crates/core/src/callbacks.rs — add:
/// Locate `callback_name` in `callbacks_json` and run full structural validation
/// on that single entry. Cheaper than `parse_callbacks` when the dispatcher only
/// cares about one callback per delivery row.
pub fn validate_named_callback(
    callbacks_json: &serde_json::Value,
    callback_name: &str,
    config: &DispatchConfig,
) -> Result<CallbackTarget, String> {
    let array = callbacks_json
        .as_array()
        .ok_or_else(|| "top-level callbacks value is not a JSON array".to_string())?;
    if array.len() > config.max_callbacks_per_event as usize {
        return Err(format!(
            "too many callbacks in event ({}); max is {}",
            array.len(),
            config.max_callbacks_per_event
        ));
    }
    let (idx, element) = array
        .iter()
        .enumerate()
        .find(|(_, v)| v.get("name").and_then(|n| n.as_str()) == Some(callback_name))
        .ok_or_else(|| format!("callback '{callback_name}' not found"))?;
    parse_one_callback(element, idx, config).map(|t| t).map_err(|(_, r)| r)
}
```

Then in `repo.rs`:

```rust
fn extract_callback_target(
    callbacks: &serde_json::Value,
    callback_name: &str,
    config: &DispatchConfig,
) -> Result<CallbackTarget> {
    crate::callbacks::validate_named_callback(callbacks, callback_name, config)
        .map_err(|reason| crate::error::Error::InvalidData(format!(
            "callback '{callback_name}' failed structural revalidation at dispatch time: {reason}"
        )))
}
```

**Why this fix**

Keeps the dispatch-time defense-in-depth (the same `parse_one_callback` runs as at schedule time) while doing O(N) work per delivery instead of O(N²) work and discarding nothing. Also makes the dispatch path's intent clearer: "validate this one callback against current config", not "parse every callback the publisher sent".

### Finding 3 — Array-level rejections surface as misleading "callback not found"

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/repo.rs:740-757` |
| **Severity** | Low |
| **Category** | Correctness |

**Problem**

When `parse_callbacks` rejects the whole array (because `array.len() > config.max_callbacks_per_event` or the value isn't a JSON object array), it pushes a single entry into `invalid` keyed by the synthetic name `<root>` and returns an empty `valid`. `extract_callback_target` then:

1. Fails to find `callback_name` in `parsed.invalid` (it's looking up `"send_email"`, `<root>` doesn't match).
2. Falls through to `parsed.valid.into_iter().find(...)`, which is empty, and emits `CallbackTargetMissing("callback 'send_email' not found in event callbacks array")`.

Realistic trigger: an operator lowers `dispatch.max_callbacks_per_event` after events are already scheduled. The dead-letter `last_error` field then says the callback is "not found" even though the row's JSON clearly contains it — operators chase the wrong bug.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/repo.rs lines 740-757
    let parsed = crate::callbacks::parse_callbacks(callbacks, config);

    // If the named callback appears in `invalid`, it failed structural revalidation.
    if let Some((_, reason)) = parsed.invalid.iter().find(|(n, _)| n == callback_name) {
        return Err(crate::error::Error::InvalidData(format!(
            "callback '{callback_name}' failed structural revalidation at dispatch time: {reason}"
        )));
    }

    parsed
        .valid
        .into_iter()
        .find(|t| t.name == callback_name)
        .ok_or_else(|| {
            crate::error::Error::CallbackTargetMissing(format!(
                "callback '{callback_name}' not found in event callbacks array"
            ))
        })
```

**Recommended fix**

Treat any `<root>`-keyed rejection as a structural validation failure, not as "callback missing", so operators see the real cause.

```rust
    let parsed = crate::callbacks::parse_callbacks(callbacks, config);

    if let Some((_, reason)) = parsed.invalid.iter().find(|(n, _)| n == callback_name) {
        return Err(crate::error::Error::InvalidData(format!(
            "callback '{callback_name}' failed structural revalidation at dispatch time: {reason}"
        )));
    }
    if let Some((_, reason)) = parsed.invalid.iter().find(|(n, _)| n == "<root>") {
        return Err(crate::error::Error::InvalidData(format!(
            "callbacks array failed structural revalidation at dispatch time: {reason}"
        )));
    }

    parsed
        .valid
        .into_iter()
        .find(|t| t.name == callback_name)
        .ok_or_else(|| {
            crate::error::Error::CallbackTargetMissing(format!(
                "callback '{callback_name}' not found in event callbacks array"
            ))
        })
```

If Finding 2 is also taken, this collapses naturally: `validate_named_callback` returns the array-level error directly, and the calling site only has one error path.

**Why this fix**

`last_error` is the operator's only post-mortem source for dead-lettered deliveries. A misleading message ("not found") sends them inspecting JSON they already know contains the row, when the real cause is a config change or row mutation that put the array over the limit.

### Finding 4 — `#[serde(default)]` is inconsistent across `DispatchSettings` allow-flags

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/config.rs:135-155` |
| **Severity** | Low |
| **Category** | Config |

**Problem**

Inside the same `DispatchSettings` struct, four "newer" boolean/limit fields have mixed default behaviour:

- `allow_insecure_urls`: required (no `#[serde(default)]`)
- `allow_unsigned_callbacks`: required
- `allow_private_ip_targets`: `#[serde(default)]`
- `max_callbacks_per_event`: `#[serde(default = "default_max_callbacks_per_event")]`

A user upgrading from a config written before any of these were introduced gets a hard "missing field" error for the first two and a silent default for the last two. The asymmetry isn't justified by safety (all four are security-relevant: defaulting `allow_private_ip_targets = false` is safe; defaulting `allow_insecure_urls = false` would also be safe). The `[dispatch]` block in `envs/app_config.toml` sets every one of them explicitly, so there's no operational reason to make some required.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/config.rs lines 133-155
#[derive(Debug, Clone, serde::Deserialize)]
#[serde(deny_unknown_fields)]
pub struct DispatchSettings {
    pub poll_interval_secs: u64,
    pub batch_size: i64,
    pub max_attempts: u32,
    pub backoff_secs: Vec<u64>,
    pub handler_timeout_secs: u64,
    pub lock_buffer_secs: u64,
    pub external_timeout_sweep_interval_secs: u64,
    pub max_completion_cycles: u32,
    pub payload_size_limit_bytes: i64,
    pub notify_channel: String,
    pub allow_insecure_urls: bool,
    pub allow_unsigned_callbacks: bool,
    /// If true, allows callback URLs that target private/loopback IP addresses.
    /// Off by default; only enable in dev/integration-test environments.
    #[serde(default)]
    pub allow_private_ip_targets: bool,
    /// Maximum number of callbacks allowed per event. Events with more are immediately dead-lettered.
    #[serde(default = "default_max_callbacks_per_event")]
    pub max_callbacks_per_event: u32,
}
```

**Recommended fix**

Pick one policy and apply it to all four. The "secure-default" policy reads cleanest:

```rust
#[derive(Debug, Clone, serde::Deserialize)]
#[serde(deny_unknown_fields)]
pub struct DispatchSettings {
    pub poll_interval_secs: u64,
    pub batch_size: i64,
    pub max_attempts: u32,
    pub backoff_secs: Vec<u64>,
    pub handler_timeout_secs: u64,
    pub lock_buffer_secs: u64,
    pub external_timeout_sweep_interval_secs: u64,
    pub max_completion_cycles: u32,
    pub payload_size_limit_bytes: i64,
    pub notify_channel: String,
    #[serde(default)]
    pub allow_insecure_urls: bool,
    #[serde(default)]
    pub allow_unsigned_callbacks: bool,
    #[serde(default)]
    pub allow_private_ip_targets: bool,
    #[serde(default = "default_max_callbacks_per_event")]
    pub max_callbacks_per_event: u32,
}
```

Update the existing `test_log_format_pretty` and `test_database_config_acquire_timeout_secs_default` test fixtures (which currently omit `allow_private_ip_targets` / `max_callbacks_per_event` to exercise the defaults) to confirm the same holds for the other two flags.

**Why this fix**

Removes a footgun for config upgrades — adding a new opt-in security flag should be a safe additive change, not a hard parse error. All three allow-flags are semantically symmetric (each opens a security guard), so they should behave the same way at deserialization.
