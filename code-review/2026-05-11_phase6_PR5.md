### Finding 1 — Broken keyset pagination on `/v1/external-pending`

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/repo.rs:575-614`, `crates/admin-api/src/routes.rs:164-186` |
| **Severity** | High |
| **Category** | Correctness |

**Problem**

The query orders by `dispatched_at` ASC but the keyset cursor filters by `d.id < $1`, and the route returns `items.last().delivery_id` as `next_cursor`. ORDER-BY and the cursor predicate must use the same column (with a unique tiebreaker) for keyset pagination to be sound. As written, the second page can skip rows whose id is greater than the cursor's id but whose dispatched_at is later, and re-deliver rows whose id is smaller but whose dispatched_at is also smaller — i.e. operators viewing "external pending" through paged calls will see an inconsistent, partial view.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/repo.rs:575-614
async fn list_external_pending(
    &self,
    page: PageParams,
    older_than: Option<Duration>,
) -> Result<Vec<ExternalPendingRow>> {
    let older_than_secs = older_than.map(|d| d.as_secs_f64());
    let rows = sqlx::query_as!(
        ExternalPendingRow,
        r#"
        SELECT
            d.id           AS delivery_id,
            ...
        FROM outbox_deliveries d
        JOIN outbox_events e USING (event_id)
        WHERE d.completion_mode = 'external'
          AND d.dispatched_at  IS NOT NULL
          AND d.processed_at   IS NULL
          AND d.dead_letter    = FALSE
          AND ($1::bigint IS NULL OR d.id < $1)
          AND ($2::text   IS NULL OR d.callback_name = $2)
          AND ($3::float8 IS NULL
               OR EXTRACT(EPOCH FROM (now() - d.dispatched_at)) > $3)
        ORDER BY d.dispatched_at
        LIMIT $4
        "#,
        page.cursor,
        page.callback_name,
        older_than_secs,
        page.limit,
    )
    .fetch_all(&self.pool)
    .await?;
    Ok(rows)
}
```

**Recommended fix**

Pick **one** axis and use it for both ORDER BY and the cursor predicate. The simplest, deterministic fix is to paginate by `d.id` (unique, monotonic) consistently with `list_dead_letters`:

```rust
async fn list_external_pending(
    &self,
    page: PageParams,
    older_than: Option<Duration>,
) -> Result<Vec<ExternalPendingRow>> {
    let older_than_secs = older_than.map(|d| d.as_secs_f64());
    let rows = sqlx::query_as!(
        ExternalPendingRow,
        r#"
        SELECT
            d.id           AS delivery_id,
            d.event_id,
            e.kind         AS event_kind,
            d.callback_name,
            d.attempts,
            d.dispatched_at AS "dispatched_at!",
            d.created_at
        FROM outbox_deliveries d
        JOIN outbox_events e USING (event_id)
        WHERE d.completion_mode = 'external'
          AND d.dispatched_at  IS NOT NULL
          AND d.processed_at   IS NULL
          AND d.dead_letter    = FALSE
          AND ($1::bigint IS NULL OR d.id < $1)
          AND ($2::text   IS NULL OR d.callback_name = $2)
          AND ($3::float8 IS NULL
               OR EXTRACT(EPOCH FROM (now() - d.dispatched_at)) > $3)
        ORDER BY d.id DESC
        LIMIT $4
        "#,
        page.cursor,
        page.callback_name,
        older_than_secs,
        page.limit,
    )
    .fetch_all(&self.pool)
    .await?;
    Ok(rows)
}
```

If sort-by-dispatched_at is required for the operator workflow, switch the cursor to `(dispatched_at, id)`: change `PageParams` (or add a separate cursor type) to carry both, predicate `WHERE (d.dispatched_at, d.id) > ($1, $2)`, and emit a compound `next_cursor` from the route.

**Why this fix**

Keyset pagination is only stable when the WHERE-predicate columns match the ORDER-BY columns; otherwise concurrent INSERTs and ties cause silent skip/duplicate. Aligning on `d.id DESC` (matching `list_dead_letters`) is the cheapest fix and uses the existing BIGSERIAL index.

Also add an integration test (testcontainers) that inserts more than `limit` external-pending rows and verifies page-by-page traversal returns every row exactly once.

### Finding 2 — `retry_delivery` has no `locked_until` guard

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/repo.rs:103-109` (trait doc), `crates/core/src/repo.rs:616-636` (impl), `crates/admin-api/src/routes.rs:207-213` (route) |
| **Severity** | High |
| **Category** | Concurrency |

**Problem**

The `Repo` trait doc on `retry_delivery` explicitly states the gap and defers the fix to Phase 6 ("A `locked_until` guard will be added when the admin API is implemented in Phase 6"). Phase 6 ships without that guard, and the admin route calls `repo.retry_delivery(id)` directly. Sequence:

1. Dispatcher locks delivery #42 (`locked_until = now() + 30 s`) and starts the HTTP call.
2. Operator calls `POST /v1/deliveries/42/retry`.
3. `retry_delivery` sets `locked_until = NULL`, `available_at = now()`, `dispatched_at = NULL`, `attempts = 0`.
4. The next scheduler cycle picks #42 up because it now matches "due and unlocked".
5. Two HTTP requests for the same delivery are in flight; the receiver sees a duplicate that bypasses the `locked_until` contract (§3.4) the rest of the system relies on.

This violates the "`locked_until` prevents duplicate dispatch across replicas" invariant called out in CLAUDE.md.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/repo.rs:616-636
async fn retry_delivery(&self, id: i64) -> Result<bool> {
    let result = sqlx::query!(
        r#"
        UPDATE outbox_deliveries
        SET attempts          = 0,
            last_error        = NULL,
            available_at      = now(),
            dispatched_at     = NULL,
            processed_at      = NULL,
            completion_cycles = 0,
            dead_letter       = FALSE,
            locked_until      = NULL
        WHERE id = $1
        "#,
        id,
    )
    .execute(&self.pool)
    .await?;

    Ok(result.rows_affected() == 1)
}
```

**Recommended fix**

Refuse to reset a row that is currently locked, and expose that distinction to the caller so the admin API can return a 409. The minimum change in `repo.rs`:

```rust
/// Outcome of an admin retry request.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum RetryOutcome {
    /// Row reset; will be picked up by the next dispatcher cycle.
    Reset,
    /// No delivery row with this id.
    NotFound,
    /// Row is currently locked by an in-flight dispatcher; refused to avoid double-dispatch.
    Locked,
}

async fn retry_delivery(&self, id: i64) -> Result<RetryOutcome> {
    let result = sqlx::query!(
        r#"
        UPDATE outbox_deliveries
        SET attempts          = 0,
            last_error        = NULL,
            available_at      = now(),
            dispatched_at     = NULL,
            processed_at      = NULL,
            completion_cycles = 0,
            dead_letter       = FALSE,
            locked_until      = NULL
        WHERE id = $1
          AND (locked_until IS NULL OR locked_until < now())
        "#,
        id,
    )
    .execute(&self.pool)
    .await?;

    if result.rows_affected() == 1 {
        return Ok(RetryOutcome::Reset);
    }
    // Distinguish "not found" from "locked" with a single follow-up SELECT.
    let exists = sqlx::query_scalar!(
        "SELECT EXISTS(SELECT 1 FROM outbox_deliveries WHERE id = $1) AS \"exists!\"",
        id,
    )
    .fetch_one(&self.pool)
    .await?;
    Ok(if exists { RetryOutcome::Locked } else { RetryOutcome::NotFound })
}
```

And the admin route:

```rust
async fn retry_delivery(State(state): State<AdminState>, Path(id): Path<i64>) -> Response {
    match state.repo.retry_delivery(id).await {
        Ok(RetryOutcome::Reset) => Json(MutationResponse { ok: true }).into_response(),
        Ok(RetryOutcome::NotFound) => (StatusCode::NOT_FOUND, "delivery not found").into_response(),
        Ok(RetryOutcome::Locked) => (
            StatusCode::CONFLICT,
            "delivery is currently locked by a dispatcher; retry again shortly",
        )
            .into_response(),
        Err(e) => internal_error(e),
    }
}
```

Also remove the stale "will be added when the admin API is implemented in Phase 6" sentence from the trait doc once landed.

**Why this fix**

The `locked_until` invariant is the entire reason dispatch is safe across replicas; the admin path must respect it. Returning 409 (rather than blocking or silently succeeding) lets operators retry seconds later when the lock expires, and aligns with the TDD's idempotency claim ("retrying a non-dead-lettered, unprocessed delivery is a no-op") because we only refuse the *actively in-flight* case.

### Finding 3 — `/health` (and `/ready`) under bearer auth conflicts with TDD §9.3 deployment intent

| Field | Value |
|-------|-------|
| **File:Line** | `crates/admin-api/src/routes.rs:36-58` |
| **Severity** | Medium |
| **Category** | Config (deployment) |

**Problem**

TDD §9.1 says "All endpoints require `Authorization: Bearer ${ADMIN_TOKEN}`", and §9.3 says `/health` is used as the "Cloud Run min-instances liveness check" and `/ready` is used by "load balancers and during deploys." Cloud Run's container probes can be configured to send a custom header, but many load balancers and orchestrators cannot. The route file places `/health` and `/ready` behind the same `require_bearer_token` middleware, so a probe without the token will get a 401 and the platform will mark the instance unhealthy.

The implementation is literal-compliant with §9.1 but operationally at odds with §9.3.

**Context** (surrounding code as it exists today)

```rust
// crates/admin-api/src/routes.rs:36-58
pub fn build_router(state: AdminState, auth_token: String) -> Router {
    Router::new()
        // Health and readiness — protected by the same bearer token for consistency,
        // but lightweight (no db hit for /health).
        .route("/health", get(health))
        .route("/ready", get(ready))
        // Dead letters
        .route("/v1/dead-letters", get(list_dead_letters))
        // External pending
        .route("/v1/external-pending", get(list_external_pending))
        // Event detail
        .route("/v1/events/{event_id}", get(get_event))
        // Delivery mutations
        .route("/v1/deliveries/{id}/retry", post(retry_delivery))
        .route("/v1/deliveries/{id}/complete", post(complete_delivery))
        .route("/v1/deliveries/{id}/abandon", post(abandon_delivery))
        // Stats
        .route("/v1/stats", get(get_stats))
        // Auth middleware wraps all routes.
        .layer(middleware::from_fn(require_bearer_token))
        .layer(Extension(ExpectedToken(auth_token)))
        .with_state(state)
}
```

**Recommended fix**

Split the router so `/health` and `/ready` are merged in *outside* the auth layer. Decide together with the operator on whether `/ready` stays protected — keep `/health` unauthenticated unconditionally so it works as a liveness probe everywhere.

```rust
pub fn build_router(state: AdminState, auth_token: String) -> Router {
    let public = Router::new()
        .route("/health", get(health))
        .route("/ready", get(ready))
        .with_state(state.clone());

    let protected = Router::new()
        .route("/v1/dead-letters", get(list_dead_letters))
        .route("/v1/external-pending", get(list_external_pending))
        .route("/v1/events/{event_id}", get(get_event))
        .route("/v1/deliveries/{id}/retry", post(retry_delivery))
        .route("/v1/deliveries/{id}/complete", post(complete_delivery))
        .route("/v1/deliveries/{id}/abandon", post(abandon_delivery))
        .route("/v1/stats", get(get_stats))
        .layer(middleware::from_fn(require_bearer_token))
        .layer(Extension(ExpectedToken(auth_token)))
        .with_state(state);

    public.merge(protected)
}
```

Update §9.3 of the TDD (or add a comment in `build_router`) to reflect the chosen exception. Also revise the auth tests in `routes.rs` (`health_requires_auth`, `health_returns_200_with_valid_token`) accordingly.

**Why this fix**

A liveness probe that 401s defeats its purpose. `/health` is a constant string and `/ready` only leaks coarse cluster health signals; the security cost of exposing them is minimal compared with the operational cost of unreachable probes.

### Finding 4 — Bearer scheme matched case-sensitively

| Field | Value |
|-------|-------|
| **File:Line** | `crates/admin-api/src/auth.rs:13-32` |
| **Severity** | Low |
| **Category** | Idiom |

**Problem**

`.strip_prefix("Bearer ")` requires an exact match of the scheme. RFC 6750 §2.1 makes the credential scheme name case-insensitive (`bearer`, `BEARER`, `Bearer` are equivalent). For an internal admin API the impact is bounded — operators control the client — but the constraint is invisible from the response and shows up as a confusing 401.

**Context** (surrounding code as it exists today)

```rust
// crates/admin-api/src/auth.rs:13-32
pub async fn require_bearer_token(request: Request, next: Next) -> Response {
    let expected: &ExpectedToken = request
        .extensions()
        .get::<ExpectedToken>()
        .expect("ExpectedToken extension must be set by the router");

    let authorised = request
        .headers()
        .get(axum::http::header::AUTHORIZATION)
        .and_then(|v| v.to_str().ok())
        .and_then(|s| s.strip_prefix("Bearer "))
        .map(|provided| {
            let provided_bytes = provided.as_bytes();
            let expected_bytes = expected.0.as_bytes();
            bool::from(provided_bytes.ct_eq(expected_bytes))
        })
        .unwrap_or(false);
```

**Recommended fix**

Split on the first space and case-insensitively compare the scheme:

```rust
let authorised = request
    .headers()
    .get(axum::http::header::AUTHORIZATION)
    .and_then(|v| v.to_str().ok())
    .and_then(|s| s.split_once(' '))
    .filter(|(scheme, _)| scheme.eq_ignore_ascii_case("Bearer"))
    .map(|(_, provided)| {
        let provided_bytes = provided.as_bytes();
        let expected_bytes = expected.0.as_bytes();
        bool::from(provided_bytes.ct_eq(expected_bytes))
    })
    .unwrap_or(false);
```

**Why this fix**

Cheap to do, removes a footgun, and brings the implementation in line with RFC 6750.

### Finding 5 — `older_than` query param accepts a `f64` of seconds, not a duration string

| Field | Value |
|-------|-------|
| **File:Line** | `crates/admin-api/src/routes.rs:147-175` |
| **Severity** | Low |
| **Category** | Config (API contract) |

**Problem**

The TDD URL example shows `GET /v1/external-pending?older_than=1h&...`. The implementation takes `older_than_secs: Option<f64>`, so an operator typing `?older_than=1h` gets a 400 from axum's deserialization. Either the implementation should accept a `humantime`-style string, or the TDD should be updated to document the seconds-as-float contract. Either way the doc and the code currently disagree.

**Context** (surrounding code as it exists today)

```rust
// crates/admin-api/src/routes.rs:147-175
#[derive(Deserialize)]
struct ExternalPendingQuery {
    callback_name: Option<String>,
    /// Only include rows where `now() - dispatched_at > older_than_secs` seconds.
    older_than_secs: Option<f64>,
    /// Keyset pagination cursor (delivery_id).
    before: Option<i64>,
    #[serde(default = "default_limit")]
    limit: i64,
}
```

**Recommended fix**

Pick one of:

1. **Keep the seconds-float contract** — rename the param to make it self-describing (`older_than_secs`), update the TDD example to `?older_than_secs=3600`.
2. **Accept a duration string** — add a custom `Deserialize` that parses `1h`, `30m`, `2d`, etc. via the `humantime` crate (workspace dep already used elsewhere is `chrono`; `humantime` is a small additional dep).

If you go with option 2:

```rust
#[derive(Deserialize)]
struct ExternalPendingQuery {
    callback_name: Option<String>,
    #[serde(default, deserialize_with = "deserialize_duration_opt")]
    older_than: Option<Duration>,
    before: Option<i64>,
    #[serde(default = "default_limit")]
    limit: i64,
}

fn deserialize_duration_opt<'de, D>(d: D) -> Result<Option<Duration>, D::Error>
where
    D: serde::Deserializer<'de>,
{
    let s = Option::<String>::deserialize(d)?;
    s.map(|s| humantime::parse_duration(&s).map_err(serde::de::Error::custom))
        .transpose()
}
```

**Why this fix**

The TDD is the contract operators read; the URL example shipping a value that fails to parse will surprise the first person who tries it.

### Finding 7 — Spawned admin server task panics on runtime errors instead of triggering shutdown

| Field | Value |
|-------|-------|
| **File:Line** | `crates/bin/src/main.rs:171-185` |
| **Severity** | Low |
| **Category** | Correctness |

**Problem**

The admin-server task is spawned and the result of `axum::serve(...)` is `.expect("admin API server error")`. A runtime failure (network error, accept loop dying, etc.) panics the task and is then swallowed by the tokio runtime; the scheduler keeps running but the admin port is dead, so `/ready` can never recover and operators have no way to know without external monitoring. The startup-only failure mode (`bind` failure) deserves `.expect`; the runtime mode does not.

**Context** (surrounding code as it exists today)

```rust
// crates/bin/src/main.rs:171-185
tokio::spawn(async move {
    let addr: std::net::SocketAddr = admin_bind
        .parse()
        .expect("admin.bind was validated at startup");
    info!(bind = %addr, "admin API listening");
    let listener = tokio::net::TcpListener::bind(addr)
        .await
        .expect("failed to bind admin API listener");
    axum::serve(listener, admin_router)
        .with_graceful_shutdown(async move {
            admin_shutdown.cancelled().await;
        })
        .await
        .expect("admin API server error");
});
```

**Recommended fix**

Log the error and cancel the shutdown token so the rest of the process exits cleanly:

```rust
let admin_shutdown_signal = shutdown.clone();
let admin_shutdown_trigger = shutdown.clone();
tokio::spawn(async move {
    let addr: std::net::SocketAddr = admin_bind
        .parse()
        .expect("admin.bind was validated at startup");
    info!(bind = %addr, "admin API listening");
    let listener = match tokio::net::TcpListener::bind(addr).await {
        Ok(l) => l,
        Err(e) => {
            error!(error = %e, "failed to bind admin API listener; shutting down");
            admin_shutdown_trigger.cancel();
            return;
        }
    };
    if let Err(e) = axum::serve(listener, admin_router)
        .with_graceful_shutdown(async move {
            admin_shutdown_signal.cancelled().await;
        })
        .await
    {
        error!(error = %e, "admin API server error; shutting down");
        admin_shutdown_trigger.cancel();
    }
});
```

**Why this fix**

Treat the admin API as a critical dependency: when it dies, the process should fail fast so the orchestrator restarts it, rather than running headless.

### Finding 8 — `fetch_stats` fires five separate queries and tests don't exercise the failure path

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/repo.rs:724-820` |
| **Severity** | Low |
| **Category** | Performance |

**Problem**

`fetch_stats` opens five round-trips (4 scalar COUNTs + one grouped query). Under a deployment with non-trivial latency to Postgres, each `/v1/stats` call (which is also called by `/ready`) adds 5× the RTT. The grouped `outbox_deliveries` query already covers per-callback breakdowns; folding the four totals into the same aggregate would cut the round-trips to one Postgres call (plus the `events_total` scalar). Also, `routes::tests` cover empty/200 paths only — no test asserts that `Repo::fetch_stats` returning `Err` becomes a 500 via `internal_error`.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/repo.rs:724-820
async fn fetch_stats(&self) -> Result<Stats> {
    let events_total: i64 = sqlx::query_scalar!(...).fetch_one(&self.pool).await?;
    let deliveries_pending: i64 = sqlx::query_scalar!(...).fetch_one(&self.pool).await?;
    let deliveries_external_pending: i64 = sqlx::query_scalar!(...).fetch_one(&self.pool).await?;
    let deliveries_dead_lettered: i64 = sqlx::query_scalar!(...).fetch_one(&self.pool).await?;
    let oldest_pending_age_seconds: Option<f64> = sqlx::query_scalar!(...).fetch_one(&self.pool).await?;
    let rows = sqlx::query_as!(StatsRow, ...).fetch_all(&self.pool).await?;
    ...
}
```

**Recommended fix**

Consolidate the four `outbox_deliveries` aggregates into a single query that returns one row of totals plus the per-callback grouped rows in a second query. One concrete shape:

```rust
async fn fetch_stats(&self) -> Result<Stats> {
    let events_total: i64 =
        sqlx::query_scalar!(r#"SELECT COUNT(*) AS "count!" FROM outbox_events"#)
            .fetch_one(&self.pool)
            .await?;

    let totals = sqlx::query!(
        r#"
        SELECT
            COUNT(*) FILTER (
                WHERE dispatched_at IS NULL AND processed_at IS NULL AND dead_letter = FALSE
            ) AS "pending!",
            COUNT(*) FILTER (
                WHERE completion_mode = 'external'
                  AND dispatched_at IS NOT NULL
                  AND processed_at  IS NULL
                  AND dead_letter   = FALSE
            ) AS "external_pending!",
            COUNT(*) FILTER (WHERE dead_letter = TRUE) AS "dead_lettered!",
            EXTRACT(EPOCH FROM (now() - MIN(available_at) FILTER (
                WHERE dispatched_at IS NULL AND processed_at IS NULL AND dead_letter = FALSE
            )))::float8 AS "oldest_pending_age_seconds"
        FROM outbox_deliveries
        "#
    )
    .fetch_one(&self.pool)
    .await?;
    // ...one grouped query for per-callback breakdown unchanged.
}
```

Add a unit test in `routes::tests` that wires a `MockRepo` returning `Err(...)` from `fetch_stats` and asserts `/v1/stats` returns 500.

**Why this fix**

Same data, one TCP round-trip instead of five — meaningful when `/ready` is polled by load-balancers every few seconds. Re-generate `.sqlx/` cache (`cargo sqlx prepare --workspace`) afterwards.

### Finding 10 — Misleading constant-time comment in `auth.rs`

| Field | Value |
|-------|-------|
| **File:Line** | `crates/admin-api/src/auth.rs:27-31` |
| **Severity** | Low |
| **Category** | Security |

**Problem**

The middleware comment claims:

> Constant-time comparison: both slices padded to the longer length so the comparison never short-circuits on length mismatch alone.

This is false. The `subtle::ConstantTimeEq` impl for `[u8]` returns `Choice::from(0)` immediately on length mismatch — it does **not** pad. So token length is observable through timing. The token is short and a length oracle is a minor leak, but the comment misleads a future reader into believing the code defends against something it doesn't.

**Context** (surrounding code as it exists today)

```rust
// crates/admin-api/src/auth.rs:26-32
.map(|(_, provided)| {
    // Constant-time comparison: both slices padded to the longer length so
    // the comparison never short-circuits on length mismatch alone.
    let provided_bytes = provided.as_bytes();
    let expected_bytes = expected.0.as_bytes();
    bool::from(provided_bytes.ct_eq(expected_bytes))
})
```

**Recommended fix**

Either correct the comment, or actually equalise the lengths before comparing. The minimal honest fix is the comment:

```rust
.map(|(_, provided)| {
    // Constant-time compare of the per-byte token bytes via `subtle::ConstantTimeEq`.
    // Note: the slice impl short-circuits on length mismatch, so token length is not
    // protected — acceptable here because the bearer token is operator-controlled and
    // length is not considered secret.
    let provided_bytes = provided.as_bytes();
    let expected_bytes = expected.0.as_bytes();
    bool::from(provided_bytes.ct_eq(expected_bytes))
})
```

If you do want length-blind comparison, hash both sides with `Sha256` first and `ct_eq` the digests — both sides become 32 bytes and the timing is fully content-bound.

**Why this fix**

Comments that misrepresent security guarantees rot into false confidence; either match the comment to the code or the code to the comment.

### Finding 12 — `std::sync::Mutex` for `last_cycle_at` in async code

| Field | Value |
|-------|-------|
| **File:Line** | `crates/admin-api/src/routes.rs:26`, `:78-83`; `crates/core/src/scheduler.rs:49`, `:207-211`; `crates/bin/src/main.rs:146` |
| **Severity** | Low |
| **Category** | Concurrency |

**Problem**

CLAUDE.md is explicit: "Use `tokio::sync` primitives in async code, not `std::sync::Mutex` / `RwLock`." `LastCycleAt = Arc<std::sync::Mutex<Option<Instant>>>` is used inside async route handlers and inside the scheduler loop. The critical sections are tiny and not held across `.await`, so this is **safe** today — but it sets a pattern that's a footgun if anyone later adds an `.await` inside the guard, and it contradicts the project convention.

**Context** (surrounding code as it exists today)

```rust
// crates/admin-api/src/routes.rs:23-29
pub struct AdminState {
    pub repo: Arc<dyn Repo>,
    pub listener_status: Arc<AtomicBool>,
    pub last_cycle_at: Arc<std::sync::Mutex<Option<Instant>>>,
    pub ready_deadline: Duration,
}
```

**Recommended fix**

The state is read-mostly and trivially small. Two equivalent options:

1. **Replace with `Arc<AtomicI64>`** holding millis-since-UNIX-epoch (or 0 for "never"). Lock-free, async-safe, no risk of poisoning. Update the scheduler to write `Utc::now().timestamp_millis()` and the route to read it and compare against `Utc::now()`.

2. **Use `Arc<tokio::sync::RwLock<Option<Instant>>>`** to keep the existing shape and match the CLAUDE.md convention.

`AtomicI64` is the cleaner of the two for a single timestamp:

```rust
// scheduler.rs
pub type LastCycleAt = Arc<std::sync::atomic::AtomicI64>;

// updated each cycle
last_cycle_at.store(
    chrono::Utc::now().timestamp_millis(),
    std::sync::atomic::Ordering::Release,
);

// routes.rs
let last_cycle_ok = {
    let stored = state.last_cycle_at.load(Ordering::Acquire);
    stored != 0
        && (chrono::Utc::now().timestamp_millis() - stored)
            <= state.ready_deadline.as_millis() as i64
};
```

**Why this fix**

Aligns with the project rule, removes a potential footgun, and removes the `.unwrap()` on poisoned-mutex.
