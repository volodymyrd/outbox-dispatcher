### Finding 1 — Decoded secret bytes not zeroized when length validation fails

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/keyring.rs:144-161` |
| **Severity** | Medium |
| **Category** | Security |

**Problem**

After base64 decoding, the raw `Vec<u8>` is only wrapped in `Zeroizing` on the success path. If the secret fails the `< MIN_SECRET_BYTES` or `> MAX_SECRET_BYTES` check, `bytes` is dropped on `Vec`'s default drop, which does not zero the heap allocation. This contradicts the module-level guarantee that "secret bytes are stored in a `Zeroizing<Vec<u8>>` wrapper that overwrites the allocation with zeroes on drop." In the rejection path the bytes are still real key material — a misconfigured short secret is no less sensitive than a properly sized one.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/keyring.rs:137-163
match decoded {
    Err(e) => {
        errors.push(format!(
            "signing_keys[{id}]: failed to base64-decode secret from '{}': {e}",
            cfg.secret_env
        ));
    }
    Ok(bytes) => {
        if bytes.len() < MIN_SECRET_BYTES {
            errors.push(format!(
                "signing_keys[{id}]: secret from '{}' is {} bytes after \
                 decoding; minimum is {MIN_SECRET_BYTES}",
                cfg.secret_env,
                bytes.len()
            ));
        } else if bytes.len() > MAX_SECRET_BYTES {
            errors.push(format!(
                "signing_keys[{id}]: secret from '{}' is {} bytes after \
                 decoding; maximum is {MAX_SECRET_BYTES}",
                cfg.secret_env,
                bytes.len()
            ));
        } else {
            keys.insert(id.clone(), Zeroizing::new(bytes));
        }
    }
}
```

**Recommended fix**

```rust
Ok(bytes) => {
    let bytes = Zeroizing::new(bytes);
    if bytes.len() < MIN_SECRET_BYTES {
        errors.push(format!(
            "signing_keys[{id}]: secret from '{}' is {} bytes after \
             decoding; minimum is {MIN_SECRET_BYTES}",
            cfg.secret_env,
            bytes.len()
        ));
    } else if bytes.len() > MAX_SECRET_BYTES {
        errors.push(format!(
            "signing_keys[{id}]: secret from '{}' is {} bytes after \
             decoding; maximum is {MAX_SECRET_BYTES}",
            cfg.secret_env,
            bytes.len()
        ));
    } else {
        keys.insert(id.clone(), bytes);
    }
}
```

**Why this fix**

Wrapping in `Zeroizing` immediately after decode keeps the secret protected through every code path (including panic unwind), matching the documented invariant for the rest of the module.

### Finding 2 — `recover_cursor` query has no supporting index and may scan the events table at startup

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/repo.rs:476-493` |
| **Severity** | Medium |
| **Category** | Performance |

**Problem**

The query "find the highest `outbox_events.id` that has any delivery row" walks `outbox_events` in reverse PK order and probes `outbox_deliveries` per row. If many of the most recent events have no delivery rows yet — a normal state when the dispatcher is restarted shortly after a publisher burst, or after a long downtime where new events accumulated faster than they were scheduled — Postgres has to scan back through the gap until it finds the first event with deliveries. There is no targeted index on `outbox_deliveries(event_id)` separate from the composite UNIQUE `(event_id, callback_name)`; the planner can still use the unique index for the join, but the outer reverse scan grows unbounded with the gap size. Cursor recovery runs once at startup, so this is not a hot path, but a multi-minute startup delay on a large `outbox_events` table after an incident is a real operational concern.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/repo.rs:476-493
async fn recover_cursor(&self) -> Result<i64> {
    let cursor = sqlx::query_scalar!(
        r#"
        SELECT COALESCE(
            (SELECT e.id
             FROM outbox_deliveries d
             JOIN outbox_events e ON e.event_id = d.event_id
             ORDER BY e.id DESC
             LIMIT 1),
            0
        ) AS "cursor!"
        "#,
    )
    .fetch_one(&self.pool)
    .await?;

    Ok(cursor)
}
```

**Recommended fix**

```rust
async fn recover_cursor(&self) -> Result<i64> {
    // Ask the deliveries side first, then look up the corresponding event id.
    // outbox_deliveries(id) is the PK so MAX(id) is an O(1) index lookup;
    // the join lookup uses the FK index and runs once.
    let cursor = sqlx::query_scalar!(
        r#"
        SELECT COALESCE(
            (SELECT e.id
             FROM outbox_events e
             WHERE e.event_id = (
                 SELECT d.event_id
                 FROM outbox_deliveries d
                 ORDER BY d.id DESC
                 LIMIT 1
             )),
            0
        ) AS "cursor!"
        "#,
    )
    .fetch_one(&self.pool)
    .await?;

    Ok(cursor)
}
```

**Why this fix**

The `outbox_deliveries(id)` BIGSERIAL PK is monotonic, so the most recently inserted delivery row is the one with the highest `id`. Reading that single row and joining once on `event_id` is constant-time regardless of how many events lack deliveries.

### Finding 3 — `dispatch.max_callbacks_per_event` validation only rejects zero; no upper bound

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/config.rs:469-470` |
| **Severity** | Medium |
| **Category** | Config |

**Problem**

`AppConfig::validate` enforces lower bounds *and* upper bounds for every other dispatch limit (`max_attempts ≤ MAX_PER_CALLBACK_ATTEMPTS`, `handler_timeout_secs ≤ MAX_HANDLER_TIMEOUT_SECS`, `max_completion_cycles ≤ MAX_COMPLETION_CYCLES_LIMIT`, `backoff_secs[i] ≤ MAX_BACKOFF_ELEMENT_SECS`), but `max_callbacks_per_event` only checks `> 0`. An operator could set it to `u32::MAX` (or just an excessive value like 100_000), at which point a single misbehaving publisher event with a huge `callbacks` array would cause the scheduler to materialise tens of thousands of delivery rows and run `parse_one_callback` on each — uncapped fan-out is a DoS vector that the rest of the validation deliberately closes off.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/config.rs:469-471
if self.dispatch.max_callbacks_per_event == 0 {
    errors.push("dispatch.max_callbacks_per_event must be > 0".to_string());
}
```

**Recommended fix**

Pick a defensible ceiling (matching the existing `MAX_PER_CALLBACK_ATTEMPTS = 50` cap precedent — say 1000) and enforce it in both `callbacks.rs` and `config.rs`:

```rust
// crates/core/src/callbacks.rs (add near the other MAX_* constants)
/// Hard ceiling on per-event `max_callbacks_per_event`; also enforced by [`AppConfig::validate`].
pub const MAX_CALLBACKS_PER_EVENT_LIMIT: u32 = 1_000;
```

```rust
// crates/core/src/config.rs:469
if self.dispatch.max_callbacks_per_event == 0 {
    errors.push("dispatch.max_callbacks_per_event must be > 0".to_string());
}
if self.dispatch.max_callbacks_per_event > MAX_CALLBACKS_PER_EVENT_LIMIT {
    errors.push(format!(
        "dispatch.max_callbacks_per_event must be <= {MAX_CALLBACKS_PER_EVENT_LIMIT}"
    ));
}
```

**Why this fix**

Symmetry with the other validated bounds — every per-event limit gets a ceiling so a fat-fingered config can't open up an amplification vector. The same constant feeds back into `parse_callbacks` if you want the runtime cap to track the config cap.

### Finding 4 — Poison-pill `UPDATE` failure is silently logged; row will be re-fetched and re-fail every poll cycle

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/repo.rs:280-313` |
| **Severity** | Low |
| **Category** | Correctness |

**Problem**

When `extract_callback_target` rejects a row, the code emits a dead-letter `UPDATE` and continues even if the `UPDATE` itself fails. In the failure path the row is still due (`available_at` unchanged, `dead_letter` still `FALSE`), so the very next `fetch_due_deliveries` call will re-fetch the same row, re-fail the parse, and re-emit the same `tracing::error!` — once per poll cycle, forever, until the underlying DB issue resolves. This isn't catastrophic, but it produces a tight error-log loop and wastes a connection per cycle on the same doomed row.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/repo.rs:284-313
tracing::error!(
    delivery_id = row.delivery_id,
    event_id = %row.event_id,
    callback_name = %row.callback_name,
    error = %err,
    "dead-lettering delivery with invalid callback spec",
);
let reason_owned = format!("invalid callback spec: {err}");
let reason = truncate_at_char_boundary(&reason_owned, 4096);
if let Err(db_err) = sqlx::query!(
    r#"
    UPDATE outbox_deliveries
    SET dead_letter  = TRUE,
        last_error   = $2,
        locked_until = NULL
    WHERE id = $1
    "#,
    row.delivery_id,
    reason,
)
.execute(&self.pool)
.await
{
    tracing::error!(
        delivery_id = row.delivery_id,
        error = %db_err,
        "failed to dead-letter poison-pill delivery",
    );
}
```

**Recommended fix**

Surface the DB error to the caller so the dispatch loop can apply its outer back-off rather than spinning. If best-effort is genuinely the desired contract, at least mark the row's `available_at` forward by the standard backoff so it doesn't reappear immediately:

```rust
match sqlx::query!(
    r#"
    UPDATE outbox_deliveries
    SET dead_letter  = TRUE,
        last_error   = $2,
        locked_until = NULL
    WHERE id = $1
    "#,
    row.delivery_id,
    reason,
)
.execute(&self.pool)
.await
{
    Ok(_) => {}
    Err(db_err) => {
        tracing::error!(
            delivery_id = row.delivery_id,
            error = %db_err,
            "failed to dead-letter poison-pill delivery; aborting batch",
        );
        return Err(db_err.into());
    }
}
```

**Why this fix**

If the DB is sick enough that a single-row UPDATE fails, the rest of the batch is almost certainly broken too — propagating the error lets the scheduler back off (poll interval, supervisor restart, etc.) instead of generating a high-frequency log storm.

### Finding 5 — `KeyRing` loaded for fail-fast validation then immediately dropped

| Field | Value |
|-------|-------|
| **File:Line** | `crates/bin/src/main.rs:60-65` |
| **Severity** | Low |
| **Category** | Idiom |

**Problem**

`KeyRing::load` decodes every secret, validates them, then the `KeyRing` is dropped on the next line. When Phase 4 wires up dispatch, the secrets will be decoded a second time. This is mostly cosmetic — `Zeroizing` cleans up the discarded copy — but it's wasted work and easy to mis-extend (a future change could leave the second load uncalled, leaving the dispatcher signing with no keys at runtime even though startup "validated" them). The fail-fast objective is fine; the discard pattern is what to fix.

**Context** (surrounding code as it exists today)

```rust
// crates/bin/src/main.rs:59-65
// Fail fast if any configured signing key cannot be resolved from its env var.
if let Err(errors) = KeyRing::load(&config.signing_keys) {
    for e in &errors.0 {
        eprintln!("signing key error: {e}");
    }
    anyhow::bail!("invalid signing keys ({} error(s))", errors.0.len());
}
```

**Recommended fix**

Bind the keyring once at startup and pass it down — this also documents that the secrets stay alive for the lifetime of the dispatcher, which is what the rest of the design assumes:

```rust
let keyring = match KeyRing::load(&config.signing_keys) {
    Ok(kr) => kr,
    Err(errors) => {
        for e in &errors.0 {
            eprintln!("signing key error: {e}");
        }
        anyhow::bail!("invalid signing keys ({} error(s))", errors.0.len());
    }
};
// keyring is now ready for Phase 4 dispatch wiring; pass it to whatever owns the
// HTTP callback driver. Until then a `let _ = &keyring;` keeps the binding live.
```

**Why this fix**

One decode pass, no risk of forgetting to wire it later, and the type system enforces that the `KeyRing` outlives anything that needs it.

### Finding 6 — `LOAD_MUTEX` only serialises env-var tests within `config::tests`, not across the test binary

| Field | Value |
|-------|-------|
| **File:Line** | `crates/core/src/config.rs:1448-1514` |
| **Severity** | Low |
| **Category** | Testing |

**Problem**

`LOAD_MUTEX` is a `static Mutex<()>` private to the `config::tests` module. It correctly prevents the three `test_load_*` tests from racing with each other on `APP_CONFIG_DIR` / `DATABASE_URL`. But the env vars themselves are process-global and `cargo test` runs every test in the same `outbox-dispatcher-core` test binary as parallel threads. If any test in another module (current or future — `keyring`, `repo`, etc.) reads or writes `DATABASE_URL` or `APP_CONFIG_DIR`, the mutex won't help and the tests will flake. The `unsafe { set_var }` SAFETY comment claims "no concurrent thread touches these vars" — that claim is one PR away from being false.

**Context** (surrounding code as it exists today)

```rust
// crates/core/src/config.rs:1448-1463
use std::sync::Mutex;
static LOAD_MUTEX: Mutex<()> = Mutex::new(());

#[test]
fn test_load_reads_base_config_from_disk() {
    use std::io::Write;
    let _guard = LOAD_MUTEX.lock().unwrap_or_else(|p| p.into_inner());

    let dir = tempfile::tempdir().expect("tempdir");
    let mut f = std::fs::File::create(dir.path().join("app_config.toml")).unwrap();
    write!(f, "{}", full_toml()).unwrap();

    // SAFETY: protected by LOAD_MUTEX; no concurrent thread touches these vars.
    unsafe { std::env::set_var("APP_CONFIG_DIR", dir.path().to_str().unwrap()) };
    let cfg = AppConfig::load("local").expect("load failed");
    unsafe { std::env::remove_var("APP_CONFIG_DIR") };
    ...
}
```

**Recommended fix**

Either (a) lift the mutex to a crate-wide test helper in `lib.rs` or a new `tests/common/mod.rs` so all env-mutating tests share it, or (b) gate these tests behind a single-threaded runner. Option (a):

```rust
// crates/core/src/lib.rs
#[cfg(test)]
pub(crate) mod test_support {
    use std::sync::Mutex;
    /// Serialises every test that reads or writes process env vars across the whole crate.
    pub static ENV_MUTEX: Mutex<()> = Mutex::new(());
}

// crates/core/src/config.rs (and any other module that touches env vars)
let _guard = crate::test_support::ENV_MUTEX
    .lock()
    .unwrap_or_else(|p| p.into_inner());
```

Or (b), if you'd rather not couple modules:

```toml
# Cargo.toml [test] target — run env-touching tests serially.
# (Or add a `serial_test` dev-dep and tag the relevant tests with `#[serial]`.)
```

**Why this fix**

Process env vars are global; the synchronisation primitive that guards them must be global too. A single shared mutex (or `serial_test`) protects future test modules from accidental flakiness without anyone needing to remember the convention.
